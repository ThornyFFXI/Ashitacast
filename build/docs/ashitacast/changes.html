If you just want to quickly convert your XMLs, you should read quickstart.html and use the import function.

 

Overview

Ashitacast for Ashita4(ACv4) is a ground-up rewrite of Ashitacast.  This means some features have changed.  Your XML will not work as is.  If you find this upsetting or frightening, don't worry!  The plugin LegacyAC is also available, and functions as an exact copy of Ashitacast for Ashita3(ACv3).  Your XML can be used as is; you'll simply need to keep it in Ashita/config/LegacyAC instead.  That said, I highly recommend anyone who maintains their own XML look into ACv4.  There are many new features to make your life easier.  Further, LegacyAC will receive a bare minimum of maintenance going forward, and will not receive any new features.  I will not personally give any support for ACv3 XMLs.  This document serves to detail all of the changes made compared to ACv3, as a reference for users who are already quite familiar with ACv3.  The other documents included serve as full documentation, so if you're starting fresh, you'll want to review them as well.

 

Top Level Node Changes

·       <settings> has been adjusted significantly.  See XML structure documentation for new options.

·       <init> has been renamed to <load> for consistancy with ashita API.

·       <unload> has been added as a companion to <load>.

·       <inputcommands> has been renamed to <commands> for brevity.

·       <idlegear> has been renamed to <default> to reduce the confusion with using 'idle gear' for engaged sets.

·       <item> has been added, and will trigger upon usage of items.

·       <premagic> has been changed to <precast> to better fit player language.

·       <midmagic> has been changed to <midcast> to better fit player language.

·       <jobability> has been changed to <ability> for brevity.

·       <variables> has been removed.  You should now use <setvar> tags within your <load> section to set initial variable states.

·       <include> tag, as previously used for dressme/packer, is now <packer>.

 

Lower Level Node Changes

·       <setvar> now supports ^ \* / + and - within the value field.  For example, you can use <setvar name="Count" value="$count + 1" /> to increase a variable by 1.

·       <incvar> and <decvar> have been removed.

·       <clearvar> has been added, with attributes of name and all.  For example, you can use <clearvar name="Count" /> to remove one variable, or <clearvar all="true" /> to clear all.

·       <clearvars> has been removed.

·       <command> now has the attribute 'mode' to determine how the game should process the command.  For example, you can use <command mode="0">/ma "Stun" $t_id</command> to specify a menu style command and act on something by ID.

·       <addtochat> and <addtoecho> have been removed.  <echo> has been added, with tags of 'mode' and 'delay' used identically to command.  For example, you can use <echo mode="0" delay="1000">A full second has passed!</echo> to add something to chat in say mode, with a 1000 millisecond delay.

·       <gearlock> now uses milliseconds instead of seconds.

·       <change>, <registerbuff>, and <clearbuff> can now accept a name attribute in place of an ID attribute.  If both are specified, ID will take precedence.

·       <doidlegear> has been removed.  It served no purpose as default is constantly parsed anyway.  Simply update variables accordingly to create the changes you want in the next pass through <default>.

·       In both <set> and <equip> tags, <lear>, <rear>, <lring>, and <rring> tags are no longer supported.  Please use <ring1>, <ring2>, <ear1>, and <ear2> for standardization.

 

Variable System Changes

·       Variables are now all referenced using a $ icon, regardless of whether they are user variables or included in ashitacast.

·       User variables will be prefaced with u_ to indicate such.  For example, after using <setvar name="Count" value="3" /> or /ac setvar Count 3, you could later use $u_Count to retrieve the value 3.

·       To reference an existing rule, such as p_name, you would simply use $p_name.

·       Variables are no longer implemented across every tag and attribute in your XML, to reduce excessive string reparsing.

 

Variable Usage - List of Valid Locations

·       Any attribute besides mode in an if or elseif tag.

·       The body of a command or echo tag.  [NOT the attributes(mode/delay).]

·       The value attribute of a setvar tag or /ac setvar command. [NOT the name attribute.]

·       The augment and/or set attribute of an equip tag. [NOT augpath, augtrial, or augrank.]

·       The name attribute of a change tag. [NOT id]        

·       The body of an equip tag.

·       Anywhere in an xml-bound command, after the command itself.

·       For example, if you have <command input=”/mycommand”>, anything after /mycommand can include variables.

·       You may also use the /ac eval command to send a raw command with variables replaced.

·       For example, if you have a monster with an index of 100 targeted and type /ac eval /echo $t_index, AC will send the command /echo 100.

 

If/Elseif Attribute Changes

·       You may name attributes after user variables now.  For example, you can use <if u_Count="3"> to directly reference a user variable named Count without needing advanced tags.

·       When using any tag other than advanced or buffactive, you may use: <= >= < > or ! symbol as the beginning of a comparand to change the evaluation.

·       If none is specified, the default evaluation will be =.

·       In tags other than advanced, you may seperate multiple comparands with & or | but not both in the same tag due to ambiguity.  For more complex comparisons, please use advanced tag.

For example, <if p_mp="<500|>1000|=520"> is a valid tag because it only uses |.  It will be functionally split into:  ($p_mp<500)  ($p_mp>1000) ($p_mp=520) and resolve true if any of the 3 conditions are true.

However,  <if p_mp="<500|>1000&<2000">] is not a valid tag because it uses both | and &.  To accomplish this, you would need to use an advanced tag: <if p_mp="<500|>1000" advanced="$p_mp<2000">.

·       You may use an asterisk(\*) to indicate a wildcard when comparing strings in basic tags, but not in the advanced tag.

For example, if you’re casting Reraise and you have this rule: <if ad_name=”*raise”>, it will resolve true.  However, <if advanced=”$ad_name=*raise”> will resolve false, as advanced tags cannot use wildcards.

 

Advanced Tag Changes

·       Advanced tags now use order of operations, and only honor parentheses () not other forms of braces.

·       Advanced tags will be evaluated in the order (), ^, */, +-, <>!=, |&.  You may layer multiple parentheses as needed.

·       Advanced tags cannot use wildcards.

·       | and & will no longer honor positive values as true the way a C language would.  Only expressions that have evaluated to true are treated as true.

 

Buff Check Changes

Buffactive has been removed.  Instead, use b_id or b_name to receive a count.

For example, $b_33 will return the amount of buffs with ID of 33 you have active(likely 0 or 1, as it is haste.)

Similarly, $b_haste will return the amount of buffs with the name haste you have active(whether they are id 33(haste spell) or id 580(indi-haste effect).

As an example usage, <if b_33=">0"> and <if b_haste=">0"> will both evaluate as true if you have the haste buff active.

However, <if b_haste=">0"> will also evaluate as true if you have indi-haste active, even if you do not have the haste buff active.

To use buff checks in advanced rules, simply include the $ prefix like any other variable, such as: <if advanced="$b_33>0">.

Don't forget that you can use comments when working off IDs to save yourself confusion later: <if b_33=">0"> <!--Buff 33 is haste spell!-->

 

Augment Changes

Augment codes are no longer in use.

The attribute augpath has been added.  This will check the path on Delve, Escha, Dynamis, or any other gear with a RP system.  For example, <body augpath="B">Adhemar Jacket +1</body> will equip a path B adhemar Jacket +1.

The attribute augrank has been added.  This will check the rank on Delve, Escha, Dynamis, or any other gear with a RP system.   For example, <main augrank="25">Rostam</main> will equip a rank 25 Rostam.

The attribute augtrial has been added.  This will check the currently flagged trial on any magian gear.  For example, <main augtrial="71">Mantis</main> will equip a Mantis with the trial #71 flagged.

The attribute augment has been added.  This will check string based augments on dynamically augmented pieces(oseem, FoV, refractive capes, etc.).  To list multiple augments, separate them with commas.

For example, <head augment="'Fast Cast'+7">Merlinic Hood</head> will only equip a Merlinic Hood that has a Fast Cast+7 augment(whether or not it has any other augments).

For another example, <feet augment="'Triple Atk.'+4,Accuracy+20">Herculean Boots</feet> will only equip a Herculean Boots that has both Triple Atk.+4 and Accuracy+20 augments.

 

Note that the augment attribute will not recognize evolith augments, or some path based augments.  Whenever possible, path based augments should be specified using augpath.  Note also that you should replace " with ' for XML compliance.  You may list multiple augments, seperated by commas.  These tags prevent equipping items that do not have the parameters listed, but are not necessary.  If you only have one of an item, you don't need to use any augment tags to ensure it is the one equipped, even if it has augments.  Further, string augments in the augment attribute do not need to include every augment on the item.  But, every string augment listed must match exactly.  If in doubt when constructing augment tags, I highly recommend using addset to ensure their accuracy.  Addset will use augpath attribute for items with paths, and augment attribute for other augmented items.  When data is available, augpath items will have their augments listed as a comment.

 

Command Changes

·       /ac addset will now replace existing sets, allowing you to adjust your sets on the fly without any text editor.

·       /ac set now interprets duration in milliseconds instead of seconds.

·       /ac validate and /ac gear are not currently implemented, as they require Packer.  They will be added when Packer is functional for Ashita4.

·       /ac action has been removed, as injected packets are viewable in Ashita4, which removes the necessity for it.

·       /ac override has been removed, and will be handled in the configuration section of your XML.

·       /ac setjob and /ac setsub have been removed, as injected job change packets can be detected in Ashita4.

·       /ac var has been broken into /ac setvar, /ac clearvar, and /ac clearvars.

·       /ac import has been added, allowing you to automatically convert most ACv3 XMLs.

 

Misc. Changes

·       ACv4 will correctly insert swaps for injected action packets.

·       ACv4 will detect job change when caused by injected packets.

·       ACv4 will report the line and type of error when XML parsing fails.

·       ACv4 will report various misuses of tags, including elseif, else, and change that would simply be ignored in ACv3.

 

*New* Include Tags

Include tags have been added.  A basic include tag will simply be replaced with the entire contents of the listed file.  You may list a relative path to the current file, or an absolute path.  If no path is listed, the folder the current XML is in will be checked.  For example, you have an XML, derp_rdm.xml:

<ashitacast>

          <sets>

                   <set name="precast">

                             <head>Merlinic Hood</head>

                   </set>

                   <set name="midcast">

                             <head>Amalric Coif +1</head>

                   </set>

          </sets>

<include>rdm_rules.xml</include>

</ashitacast>

 

Then, in the same folder, you have rdm_rules.xml:

          <precast>

                   <equip set="precast" />

                   <if ad_name="Impact"><equip><body>Twilight Cloak</body></equip></if>

          </precast>

          <midcast>

                   <equip set="midcast" />

                   <if ad_name="Impact"><equip><body>Twilight Cloak</body></equip></if>

          </midcast>

                  

A basic include tag simply replaces it, resulting in:

<ashitacast>

          <sets>

                   <set name="precast">

                             <head>Merlinic Hood</head>

                   </set>

                   <set name="midcast">

                             <head>Amalric Coif +1</head>

                   </set>

          </sets>

          <precast>

                   <equip set="precast" />

                   <if ad_name="Impact"><equip><body>Twilight Cloak</body></equip></if>

          </precast>

          <midcast>

                   <equip set="midcast" />

                   <if ad_name="Impact"><equip><body>Twilight Cloak</body></equip></if>

          </midcast>

</ashitacast>

 

Self-merging include tags have also been added.  These will combine all top level nodes from the target xml into your current XML.  Top level nodes are:

·       sets

·       load

·       unload

·       commands

·       default

·       preranged

·       midranged

·       precast

·       midcast

·       ability

·       weaponskill

·       item

·       petskill

·       petspell

 

If the include tag is **before** a node, the content of that node in the included XML will be merged to the start of that node in the current XML.  If the include tag is **after** a node, the content of that node in the included XML will be merged to the end of that node in the current XML.  To reduce confusion, it is recommended that you keep any merged include files solely ‘start of section’ or ‘end of section’ content and place them immediately after your <ashitacast> tag if they are start of section, or immediately before your </ashitacast> tag if they are end of section.

 

For example, you have an XML, derp_rdm.xml:

<ashitacast>

          <precast>

                   <equip set="precast" />

          </precast>

          <midcast>

                   <equip set="midcast" />

          </midcast>                  

          <include merge="true">twilightcloak.xml</include>

</ashitacast>

 

And, the contents of twilightcloak.xml:

<ashitacast>

          <precast>

                   <if ad_name="Impact"><equip><body>Twilight Cloak</body></equip></if>

          </precast>

          <midcast>

                   <if ad_name="Impact"><equip><body>Twilight Cloak</body></equip></if>

          </midcast>

</ashitacast>

         

When ashitacast loads derp_rdm.xml, it will create a merged file that looks like this:

<ashitacast>

          <precast>

                   <equip set="precast" />

                   <if ad_name="Impact"><equip><body>Twilight Cloak</body></equip></if>

          </precast>

          <midcast>

                   <equip set="midcast" />

                   <if ad_name="Impact"><equip><body>Twilight Cloak</body></equip></if>

          </midcast>

</ashitacast>

         

Note that since the <include> was **after** the <precast> and <midcast> sections, the inserted content appeared at the end of them.  If we change derp_rdm.xml to:

<ashitacast>

          <include merge="true">twilightcloak.xml</include>

          <precast>

                   <equip set="precast" />

          </precast>

          <midcast>

                   <equip set="midcast" />

          </midcast>                  

</ashitacast>

 

The output will change to:

<ashitacast>

          <precast>

                   <if ad_name="Impact"><equip><body>Twilight Cloak</body></equip></if>

                   <equip set="precast" />

          </precast>

          <midcast>

                   <if ad_name="Impact"><equip><body>Twilight Cloak</body></equip></if>

                   <equip set="midcast" />

          </midcast>

</ashitacast>

 

Further, if we change derp_rdm.xml to:

<ashitacast>

          <precast>

                   <equip set="precast" />

          </precast>

          <include merge="true">twilightcloak.xml</include>

          <midcast>

                   <equip set="midcast" />

          </midcast>                  

</ashitacast>

 

The output will change to:

<ashitacast>

          <precast>

                   <equip set="precast" />

                   <if ad_name="Impact"><equip><body>Twilight Cloak</body></equip></if>

          </precast>

          <midcast>

                   <if ad_name="Impact"><equip><body>Twilight Cloak</body></equip></if>

                   <equip set="midcast" />

          </midcast>

</ashitacast>

 

You may use any amount of <include> tags of either variety.  Simple includes will be processed from start to end of file, then merge includes from start to end of file.  If you have GenerateMergedFiles enabled, the completely merged file after all includes will be generated in Ashita/config/ashitacast/merged/filename.xml after it has been merged and loaded into memory.  This is to allow you to more easily troubleshoot any issues that may arise due to mistakes in your include tags, and is not directly used by ashitacast for anything.  If your XML fails to parse, any line numbers or error codes will be referring to the merged file, not the original file.  Note that you will still have to correct the issues in the original file after locating them if you plan to load via original file.
